# Penpot 私有 Fork：部署与使用教程

本文面向 **normojs/penpot** 私有 fork（Penpot 2.15.x + first-class MCP / `penpot-cli`）。  
适用于：**首次自托管部署、默认管理员登录、模板按需下载、MCP/CLI 自动化**。

相关英文文档：

- [`mcp/docs/standalone-install.md`](../../mcp/docs/standalone-install.md)（CLI / MCP 安装）
- [`mcp/docs/product-positioning.md`](../../mcp/docs/product-positioning.md)（产品边界）
- [`mcp/docs/private-release-0.1.0.md`](../../mcp/docs/private-release-0.1.0.md)（私有 0.1.0 说明）

---

## 1. 产品是什么 / 不是什么

| 是 | 不是 |
| --- | --- |
| 可自托管的 Penpot **私有 fork** | 官方 Penpot SaaS / 官方 GA 功能 |
| 带 **MCP 服务 + 浏览器插件 + penpot-cli** | 默认 `npm install -g penpot-cli` 公开包 |
| 多用户自托管（需按文档配置） | 开箱即用的公网 SaaS 替代品 |

**CLI 私有版本**：`0.1.0`，标签 `cli-v0.1.0`，分发方式为 **release 压缩包**（非 npm）。

## 2. 架构一览

```text
浏览器 UI  ──►  Frontend
                 │
                 ├─► Backend (RPC / 用户 / 文件 / clone-template)
                 │
                 └─► MCP Plugin (浏览器) ◄──WS/HTTP──► MCP Server
                                                      │
CLI / Agent  ── token ──► Backend RPC                 │
CLI / Agent  ── MCP  ──► MCP Server ──────────────────┘
```

| 组件 | 典型端口（示例） | 说明 |
| --- | --- | --- |
| Frontend | `9001`（compose）/ `3449`（devenv HTTPS） | 设计器 Web UI |
| Backend | `6060`（内部） | API / 认证 / 模板导入 |
| MCP Server | `4401`（HTTP）/ `4402`（WS） | Agent 工具面 |
| MCP Plugin 预览 | `4400` | 插件 manifest / UI |
| Postgres / Redis | 内部 | 数据与实时通知 |

## 3. 部署方式选择

| 场景 | 推荐方式 |
| --- | --- |
| 给团队试用 / 接近生产 | **Docker Compose**（`docker/images/docker-compose.yaml`） |
| 开发 fork / 改后端前端 | **Devenv**（`./manage.sh start-devenv` + 容器内服务） |
| 只要脚本自动化、已有后端 | **penpot-cli 压缩包** + 运行中的 Backend/MCP |

下文以 **Compose 首次部署** 为主；开发环境见第 8 节。

## 4. 首次部署（Docker Compose）

### 4.1 前置条件

- Docker + Docker Compose
- 可访问镜像仓库与（可选）模板 CDN / 魔搭
- 建议准备：随机 `PENPOT_SECRET_KEY`

生成密钥示例：

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(64))"
```

### 4.2 配置文件

使用仓库内示例：

```bash
cd docker/images
# 按需复制/编辑 docker-compose.yaml
```

**重要：默认 `image: penpotapp/*` 是上游官方镜像，不包含本 fork 的默认管理员、魔搭模板 URI 等改动。**

推荐用本仓库脚本构建并启动 fork 栈：

```bash
# 仓库根（首次较慢：编译 backend/frontend/mcp bundle + docker build）
./scripts/build-fork-images.sh

cd docker/images
cp -n .env.fork.example .env.fork   # 编辑 SECRET / 管理员密码
docker compose --env-file .env.fork \
  -f docker-compose.yaml -f docker-compose.fork.yaml up -d
```

`docker-compose.fork.yaml` 会把镜像钉到 `local/penpot-*:dev`，并注入 `PENPOT_DEFAULT_ADMIN_*`。  
仅构建 backend：`./scripts/build-fork-images.sh backend`。

至少确认：

1. **`PENPOT_PUBLIC_URI`**：用户浏览器访问的地址（如 `http://localhost:9001`）
2. **`PENPOT_SECRET_KEY`**：生产环境必须改掉示例值
3. **`PENPOT_FLAGS`**：至少包含 `enable-login-with-password`（本仓库 compose 示例已加上）
4. **默认管理员**（推荐首次部署开启，见第 5 节；须 fork 构建的 backend）

### 4.3 启动

```bash
cd docker/images
docker compose up -d
```

等待 backend 健康后，浏览器打开 `PENPOT_PUBLIC_URI`（默认示例为 <http://localhost:9001>）。

### 4.4 验证

- 打开登录页，使用默认管理员邮箱/密码登录（若已配置）
- 后端日志中可出现类似：
  - `default admin profile created`
  - 或 `default admin already present`（重启幂等）

## 5. 首次部署默认登录账号（重要）

### 5.1 作用

在**空库首次启动**时，自动创建一个可用的登录用户，避免还要先开注册、或手动跑 REPL。

### 5.2 环境变量

| 变量 | 是否必需 | 说明 |
| --- | --- | --- |
| `PENPOT_DEFAULT_ADMIN_EMAIL` | 与密码成对 | 登录邮箱 |
| `PENPOT_DEFAULT_ADMIN_PASSWORD` | 与邮箱成对 | 登录密码，**至少 8 位** |
| `PENPOT_DEFAULT_ADMIN_FULLNAME` | 可选 | 显示名，默认 `Administrator` |

行为说明：

- **仅当邮箱 + 密码都设置**时才会创建；邮箱须为合法格式，密码 ≥ 8 位
- **幂等 / 只创建不改密**：邮箱已存在则跳过（可安全反复重启）。**修改 env 中的密码不会更新已有账号**，请在 UI 中改密
- 多副本同时首启时，并发创建冲突会当作「已存在」，**不会拖垮启动**
- 若未开启 `enable-login` / `enable-login-with-password`，账号可能建了但无法密码登录（日志会警告）
- **不会**在日志里打印密码
- 配置了默认邮箱时，会自动并入实例 **admins** 集合（`PENPOT_ADMINS`）
- **必须使用本 fork 构建的 backend 镜像**，否则环境变量被忽略

Compose 中已有注释示例（`docker/images/docker-compose.yaml`）：

```yaml
# PENPOT_DEFAULT_ADMIN_EMAIL: admin@example.com
# PENPOT_DEFAULT_ADMIN_PASSWORD: change-me-now
# PENPOT_DEFAULT_ADMIN_FULLNAME: Administrator
```

生产建议：

```bash
export PENPOT_DEFAULT_ADMIN_EMAIL='admin@your-company.com'
export PENPOT_DEFAULT_ADMIN_PASSWORD='用足够长的随机密码'
# 首次登录后立即在 UI 中修改密码
```

### 5.3 登录步骤（用户）

1. 打开 Penpot 站点登录页  
2. 使用上面配置的 **邮箱 + 密码** 登录  
3. **无需安装 git**，也无需 CLI  

### 5.4 与「演示账号 / 开放注册」的区别

| 机制 | 用途 |
| --- | --- |
| **默认管理员（本功能）** | 生产/私有部署固定首个运维账号 |
| `demo-users` + 登录页「创建演示账号」 | 开发/演示，随机邮箱密码，会过期删除 |
| `registration` | 允许自助注册 |

私有生产部署建议：**默认管理员 + 关闭或限制开放注册**。

## 6. 模板 / 库和模板（按需单文件下载）

### 6.1 原则

**使用某个模板 = HTTP 下载这一个 `.penpot` 文件**，不要整仓 `git clone` / `snapshot_download`。

用户浏览器 **不需要 git**。

### 6.2 后端流程

1. 用户在 Dashboard「库和模板」点某个模板  
2. 前端调用 `clone-template`（只带 `template-id` + 目标 `project-id`）  
3. 后端 `get-template-stream`：
   - 若本地存在 `backend/builtin-templates/<id>` → 读本地  
   - 否则对 `onboarding.edn` 中该模板的 **`:file-uri` 做 HTTP GET**（跟随重定向）  
4. 将文件流 import 进项目  

### 6.3 默认模板源（本 fork）

内置 15 个 Dashboard 模板的 `:file-uri` 指向魔搭 **单文件 resolve**，例如：

```text
https://www.modelscope.cn/models/sunlab-registry/penpot-marketplace/resolve/master/kits/builtin/welcome/kit.penpot
```

全量目录（小 JSON，可先拉目录再按 id 取 URL）：

```text
https://www.modelscope.cn/models/sunlab-registry/penpot-marketplace/resolve/master/marketplace/catalog.json
```

每条 kit 含字段 **`downloadUrl`**（已 URL 编码），业务侧只 GET 该字段即可。

### 6.4 离线 / 内网

可在 backend 工作目录预置：

```text
backend/builtin-templates/<id>
```

例如 `welcome`、`wireframing-kit`。有本地文件时**不会访问外网**。

### 6.5 错误排查

| 现象 | 常见原因 |
| --- | --- |
| `unexpected-status-code` / 302 | 旧版未跟随重定向；本 fork 已用 `req-with-redirects` |
| 超时 | 容器访问不了 GitHub/魔搭；改用本地 `builtin-templates` 或可达的 CDN |
| 模板找不到 | `template-id` 与 `onboarding.edn` 的 `:id` 不一致 |

## 7. MCP 与 penpot-cli（自动化）

### 7.1 MCP（完整栈）

MCP **不在** CLI 压缩包内，需要本 fork 部署（Compose 中的 `penpot-mcp` 或 devenv）。

典型检查：

```bash
curl -sS http://127.0.0.1:4401/status
# 期望 status=ok，registeredTools 数量 > 0
```

在 Penpot 编辑器中：

1. 打开某个文件  
2. Plugins → 安装插件 URL（开发示例：`http://127.0.0.1:4400/manifest.json`）  
3. Connect MCP  

### 7.2 penpot-cli（私有压缩包）

```bash
# 在仓库根、已装 pnpm 时打包
pnpm cli:package-check
# 产物：tmp/penpot-cli-release/penpot-cli-0.1.0.tar.gz

tar -xzf penpot-cli-0.1.0.tar.gz
cd penpot-cli-0.1.0
./bin/penpot-cli --help
```

常用环境变量：

```bash
export PENPOT_BACKEND_URI=http://127.0.0.1:6060   # 按实际 backend
export PENPOT_CLI_TOKEN='用户访问令牌'             # 设置 → 访问令牌，勿提交仓库
```

示例：

```bash
./bin/penpot-cli account me --format json
./bin/penpot-cli mcp status --url http://127.0.0.1:4401/status --format json
```

**安全：** 任何 status 工具都不应返回 raw token；日志勿打印 `PENPOT_CLI_TOKEN`。

## 8. 开发环境（Devenv）

```bash
# 仓库根
./manage.sh start-devenv
# 容器内拉起 frontend/backend/MCP 等（可用辅助脚本）
./scripts/start-full-client-devenv.sh
./scripts/start-full-client-devenv.sh --status
```

注意：

- 宿主机若占用 `4400–4403`（例如本机另一个 penpot-mcp），脚本会尝试释放端口  
- Frontend 常用：`http://localhost:3450/`；HTTPS `https://localhost:3449/` 需主机名 `localhost`（SNI）  
- render-wasm 需容器内可写 cargo 缓存；镜像钉选 Rust **1.91.0** 更稳妥  

默认管理员同样可通过容器环境变量 `PENPOT_DEFAULT_ADMIN_*` 注入（与 Compose 相同语义）。

## 9. 日常使用清单（给最终用户）

1. 打开组织提供的 Penpot 地址  
2. 使用管理员下发的 **邮箱 / 密码** 登录（或注册，取决于策略）  
3. 创建或进入团队 / 项目  
4. 在 **库和模板** 中选择模板 → 添加到项目（后台只下当前模板文件）  
5. 需要 Agent 自动化时：管理员提供 **访问令牌** + MCP 地址；用户本机用 CLI 或 MCP 客户端

## 11. 常见问题

**Q：用户电脑没装 git 能用模板吗？**  
能。模板是 **HTTP 单文件**，与 git 无关。

**Q：默认管理员每次重启会不会重复创建？**  
不会。按邮箱查重，已存在则跳过。

**Q：只配置了邮箱没配密码？**  
引导逻辑 **跳过** 创建，不会半残账号。

**Q：CLI 和完整 Penpot 是一回事吗？**  
不是。CLI 是自动化客户端；完整产品还需要 Frontend + Backend（+ 可选 MCP）。

**Q：可以宣称 1.0 / 官方 MCP 吗？**  
私有 0.1.x 请按 [`product-positioning.md`](../../mcp/docs/product-positioning.md) 表述，勿宣称官方 GA 或默认 npm。
